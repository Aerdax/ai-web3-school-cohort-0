## 笔记

### MCP 章节

**一句话定位**：MCP 解决的不是"模型会不会更聪明"，而是"模型如何以可描述、可复用、可管理的方式使用外部能力"。

**第一性原理**：模型不应该直接拥有世界；它应该通过明确协议访问被授权的上下文和工具。

#### 核心架构

MCP Server 是真正运行的进程（不是 prompt.md 描述文件），通过 JSON-RPC 2.0 通信。三类能力：

| 能力 | 说明 |
|------|------|
| Tools | 可执行函数，有副作用（核心） |
| Resources | 可读取的数据 |
| Prompts | 预定义 prompt 模板 |

#### 自然语言 → 工具调用的桥接

这是 **Agent / LLM 的能力**，不是 MCP 的能力：

```
用户："帮我整理文件夹"
  ↓
① Client 启动时调 tools/list，拿工具列表 + schema
② 工具列表 + 用户消息 → 发给 LLM
③ LLM 理解意图，输出结构化工具调用
④ Client 调 tools/call → MCP Server 真正执行
⑤ 结果回 LLM，LLM 继续决策
```

#### 分层职责

| 层 | 做什么 | 有无 AI |
|---|--------|--------|
| Agent / LLM | 理解意图、规划、决定调哪个工具 | ✅ 全部 AI |
| MCP 协议 | 标准化通信（JSON-RPC） | ❌ |
| MCP Server | 执行逻辑（API 调用、文件操作） | ❌（除非内部主动调 LLM）|

#### MCP Server 内部嵌套 AI 的情况

部分 MCP 服务内部有专有 AI 能力（如 Figma MCP 的 `generate_figma_design`），形成两层 AI 分工：

```
外层 Agent LLM（决策：什么时候调、调什么）
    ↓
MCP Server 内部 AI（领域专家：Figma 生成模型、OCR 模型）
    ↓
返回结果给外层 Agent
```

好处：封装领域复杂度、专有模型更强；代价：成本加倍、延迟更高、debug 更难。

#### 在 AI × Web3 中的位置

```
MCP 负责：工具发现 + 调用格式标准化
    ↓ 读区块浏览器 / 查合约 / 调 RPC / 生成交易草稿

Web3 账户系统负责：最终权限 + 执行边界
    ↓ 签名确认 / session key / 审计日志
```

MCP 本身不是钱包安全方案，不能替代账户权限和交易签名。两层分开设计。

#### 对主方向的启发

企业 AI Agent 里，凭证 / 文档理解可以做成带内部 AI 的 MCP：

```
Agent → invoice_mcp.parse_invoice(file)
           ↓ 内部调 OCR + 结构化提取 LLM
           ↓ 返回 {金额, 付款方, 收款方, 到期日, 异常标记}
```
外层 Agent 不关心怎么读发票，只拿结构化结果做决策。

### 代码 / 实验

实验：`experiments/mcp-readonly-server/server.py`

只读 MCP Server，暴露两个工具：

```python
# search_docs(query) — 在白名单目录内全文搜索 .md 文件，返回匹配行 + 来源路径
# get_file(path)     — 读取白名单目录内文件，返回 来源：路径 + 内容
```

**验证结果：**

| 测试 | 结果 |
|------|------|
| `tools/list` | 返回两个工具及完整 schema |
| `search_docs("MCP")` | 在 daily note 中找到 10 条，带路径和行号 |
| `get_file("daily/2026-05-29.md")` | 成功读取，返回 `来源：daily/2026-05-29.md` |
| `get_file("../../etc/passwd")` | 拦截，日志记录 `access_denied`，不静默失败 |

**运行方式：**
```bash
cd experiments/mcp-readonly-server
.venv/bin/python3 server.py
```

**这个实验验证了：**
- MCP Server 是真正运行的进程，不是描述文件
- Tool schema 让 Agent 知道怎么调（名字、参数、用途）
- 权限不在 MCP 协议层，要在代码里自己实现（白名单 + 路径检查）
- 错误要明确返回，不能静默失败

**下一步权限升级方向（章节推荐）：**
设计"写入工具"的权限升级方案：什么时候需要用户确认，如何撤销授权，如何审计每次调用。

### 卡点 & 疑问

已解决：
- MCP 不是 prompt.md 描述文件，是真正运行的服务进程
- 自然语言到工具调用的映射是 LLM 的能力，不是 MCP 的能力
- 协议统一（JSON-RPC 2.0），不同 MCP Server 客户端统一对接
- MCP Server 可以内部嵌套专有 AI 模型，形成两层 AI 分工

### 方向地图（2 分钟）

"我的主方向是**企业级 AI Agent × Web3 支付与钱包管理**。"

为什么这两个缺一不可：
- 只有 AI：业务流程可以自动化，但跨机构支付仍然依赖银行，周期长、没有链上审计
- 只有 Web3：链上支付可以编程，但业务上下文的理解、审批路由、支付条件判断，链上做不了

AI 负责：理解业务文档 → 路由审批 → 确认支付条件 → 实时查询链上状态
Web3 负责：可编程支付通道 → 多签钱包管理 → 链上审计记录

最小验证计划：三个模拟场景跑通"文档输入 → Agent 工单草稿 → 人工确认 → 链上支付"的完整链路
