## 笔记

### Handbook 阅读：提示词

**核心原则：**

| 原则 | 要点 |
|------|------|
| 控制输入输出格式 | 不依赖模型自由发挥，显式规定结构 |
| 高分线不交给 Prompt | Prompt 定方向，代码层 / 验证层兜底质量 |
| 指令分层明确 | System / User / Context / Output 各司其职 |
| 指令拆分为四段执行 | 角色 → 任务 → 约束 → 输出格式 |
| Structured Output | 下游有处理任务时必须规范化，避免解析失败 |
| Prompt Injection | Agent 接收外部输入时的安全边界，防止指令劫持 |

**Few-shot 的疑问与澄清：**

> 问：大型项目不可能给所有业务逻辑都维护示例，few-shot 是否只针对非代码场景？

澄清：Few-shot 维护的是**格式和推理模式的示例**，不是业务逻辑覆盖。核心用途：
- 告诉模型"我要的输出长这个样子"（格式示范）
- 告诉模型"遇到边界情况这样处理"（推理模式）
- 代码场景同样适用：统一错误处理风格、注释格式、函数命名规范

实践中维护 2–3 个高质量示例就够，覆盖典型结构，不需要枚举所有业务。

**Prompt Injection 的边界：**

> 问：是不是只有开发 Agent 系统才需要考虑？

基本正确。触发场景：
- Agent 接收用户自由文本输入 → 可能被注入指令
- 工具调用返回外部数据（链上数据、API 响应）→ 数据里混入指令
- 交易解释器如果加入用户自定义备注 → 也是注入面

目前阶段可以先了解概念，进入 Agent 开发时再深入防御策略。

### Handbook 阅读：上下文

**Context 设计核心：**

| 原则 | 要点 |
|------|------|
| 可刷新设计 | 记忆不能无限堆积，需要有清理 / 刷新机制 |
| 分类存储 | 持久记忆（用户偏好、权限、项目配置）vs 短暂状态（会话参数）|
| 用户输入隔离 | 用户输入与系统指令分区放置，防止越权和注入 |
| Context Window 管理 | 配合检索（RAG）、摘要、结构化数据使用；长文档按需取片段 |

**Agent Context 分层设计（5层）：**

```
┌─────────────────────────────────────────┐
│  指令层  系统规则 / 工具使用规则 / 禁止事项     │  ← 最高优先级，不可被覆盖
├─────────────────────────────────────────┤
│  任务层  用户目标 / 本次会话参数               │  ← 每次对话重置
├─────────────────────────────────────────┤
│  事实层  链上状态 / 工具结果 / simulation     │  ← 动态注入，来自工具调用
├─────────────────────────────────────────┤
│  知识层  文档 / 标准 / 论坛 / 历史案例         │  ← 检索注入，按需取
├─────────────────────────────────────────┤
│  记忆层  用户偏好 / 项目配置                  │  ← 持久化，跨会话保留
└─────────────────────────────────────────┘
```

**AI × Web3 对应关系：**
- 事实层 = 链上数据（RPC 查询结果）
- 知识层 = 合约文档、EIP 标准、协议文档
- 记忆层 = 用户钱包偏好、常用网络、风险设置

**Context Engineering / Memory / Knowledge Base：**
- 三者是 Agent 设计的核心模块，目前尚未接触过
- 优先级：先理解概念 → 再在实践中碰到时深入
