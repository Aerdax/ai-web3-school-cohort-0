### 核心概念

#### 框架（Frameworks）

**核心定位**：Framework 把模型、工具、状态、检索、评估和部署组织成可维护的系统，解决的是**工程组织问题**，不是智能本身。

**第一性原理**：先理解工作流，再决定用不用框架。先画清楚输入、状态、工具、输出、评估和失败路径，再看哪些值得交给框架。

**主要框架对比**：

| 框架 | 定位 | 适合场景 |
|------|------|---------|
| LangChain | 组件库 | 快速组合模型 + 外部系统，学习 AI 应用常见组件 |
| LangGraph | 状态机 / Graph | 多轮工具调用、重试、人工确认、分支、可恢复任务 |
| OpenAI Agents SDK | Agent 工作流 | 工具、handoff、guardrails、tracing 的结构化组织 |
| DSPy | 可优化 pipeline | 有数据集 + 评估指标 + 可重复任务，自动优化 prompt |
| Hermes | 工具调用模型生态 | 稳定产生结构化输出和 tool calling |

**判断何时需要 LangGraph**：只要开始关心任务走到哪一步、能否恢复、失败后从哪里继续，就应该考虑 graph / state machine。

**Learning Agent 安全流程**：
1. 记录失败样本 → 2. 人工标注原因 → 3. 加入 eval set → 4. 修改配置 → 5. 通过测试再上线

**AI × Web3 框架分工**：
- AI Framework：管 prompt / tools / state / eval / trace
- Web3 基础设施：管账户 / 签名 / 合约 / 链上状态
- 产品层：定义用户目标、权限边界、确认流程

### 代码 / 实验

**实验：框架对比 — bare API vs LangGraph**
路径：`experiments/framework-compare/`

同一个 Web3 tool-use 任务（查余额 / 验签名 / 查授权额度），用两种方式实现：

**bare_agent.py**（裸 API，手写 while loop）：
```python
while len(trace) < max_steps:
    response = client.chat.completions.create(model=MODEL, messages=messages, tools=TOOLS)
    msg = response.choices[0].message
    if not msg.tool_calls:
        break
    for tc in msg.tool_calls:
        result = dispatch(tc.function.name, tc.function.arguments)
        trace.append({"tool": tc.function.name, "args": ..., "result": result})
```

**graph_agent.py**（LangGraph，声明式 Graph）：
```python
class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], operator.add]
    trace: Annotated[list[dict], operator.add]

# 两个 Node + 条件 Edge，框架自动驱动循环
graph.add_conditional_edges("call_model", should_continue)
```

**compare.py 实际运行结果**（3 题，两个版本各 steps=1）：
- Q1：vitalik.eth 余额 ~5.677 ETH（真实 RPC）
- Q2：签名验证通过（本地 eth_account）
- Q3：USDC 授权额度查询成功（真实 RPC）

**四维对比结论**：

| 维度 | bare | graph | 结论 |
|------|------|-------|------|
| 可读性 | while loop 直白 | 声明式，需理解 State/Node/Edge | bare 入门快 |
| 加工具 | 改 TOOLS + 手写 dispatch | 只改 TOOLS，ToolNode 自动处理 | graph 更干净 |
| 定位错误 | 手动埋点 | State 流动可查 | graph 更易排查 |
| 写测试 | mock messages list | mock State 输入输出 | graph 边界更清晰 |

**关键感悟**：Framework 不是让代码更少，而是让复杂度**显式化**。AI × Web3 的 payment/permission 场景通常需要多步工具调用 + 可审计 trace，LangGraph 更适合。

### 卡点 & 疑问

- LangGraph `ToolNode` 返回的 `ToolMessage` 不含 args，graph trace 比 bare trace 少信息——实际生产中需要手动补全 trace 结构
- `ChatGroq` pydantic 验证要求 api_key 非空，测试时需要 `or "placeholder"` 绕过
