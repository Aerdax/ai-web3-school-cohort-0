## 二、主方向选择说明

### 主方向：企业 AI Agent × Web3 支付与钱包管理

**为什么不是纯 AI 问题？**

只用 AI：可以自动化业务流程、理解文档、路由审批、填写工单，大幅提升效率。但企业间的支付结算仍然依赖银行系统——周期长（T+1 或更慢）、跨境成本高、结算记录分散在各自系统中难以对账。当 Agent 代表企业发起支付时，传统银行接口没有为"机器发起的结构化支付请求"设计，没有标准的报价、授权、验收、结算协议。

**为什么不是纯 Web3 问题？**

只用 Web3：主流币/稳定币支付可以做到低成本、跨境、可编程结算。但企业实际的支付是有上下文的——这笔钱对应哪个合同、哪个审批、哪个业务节点，谁有权触发，触发条件是什么。链上无法自动理解业务上下文，需要 AI Agent 来桥接业务语义和链上执行。

**AI 和 Web3 同时不可替代的原因：**

- AI 负责：理解业务上下文 → 判断审批路由 → 确认支付条件满足 → 填写工单 → 实时查询链上支付状态
- Web3 负责：可编程支付通道 → 多链结算 → 链上审计记录 → 企业钱包权限管理（多签 / policy）
- 缺 AI 则业务流程仍然是人工串联；缺 Web3 则支付结算无法做到低成本、可编程、跨机构可信

## 三、问题拆解

### 参与方

| 角色 | 说明 | 痛点 |
|------|------|------|
| **业务发起方** | 触发支付或核验请求的业务人员 | 流程繁琐，需要在多个系统之间切换 |
| **审批人** | 负责确认支付/业务动作的授权人员 | 信息不聚合，判断依据分散 |
| **财务/结算方** | 负责最终支付执行的团队 | 链下结算慢、对账成本高 |
| **AI Agent** | 理解业务上下文、路由审批、查询状态、辅助决策 | 当前各平台割裂，需要统一 Agent 层 |
| **Web3 支付层** | 主流币/稳定币支付通道、企业钱包管理 | 与企业业务系统的语义鸿沟 |
| **链上审计** | 结算记录、操作日志的不可篡改存储 | 企业传统系统缺乏跨机构可信记录 |

### 流程：AI Agent 驱动的企业支付链路

```
业务事件触发（合同到期 / 发票审核通过 / 服务交付确认）
      ↓
[AI Agent] 读取业务上下文：提取关键字段、确认支付条件
      ↓
[AI Agent] 异常检测：金额异常、重复支付、条件未满足 → 标记告警
      ↓
[AI Agent] 审批路由：按金额/类型推送给对应审批人，附上摘要
      ↓
人工确认节点 ★：审批人确认意图和金额
      ↓
[AI Agent] 查询企业钱包状态：余额、Gas、链上 nonce
      ↓
[Web3] 在 policy 范围内发起支付（主流币/稳定币/多链）
      ↓
[链上] 结算完成，写入审计日志（tx hash + 业务关联 ID）
      ↓
[AI Agent] 通知相关方，更新业务系统状态
```

### 自动化边界

| 步骤 | 自动化？ | 理由 |
|------|---------|------|
| 业务上下文提取 | ✅ 自动 | 结构化提取，AI 可胜任 |
| 异常标记 | ✅ 自动（结果人工看） | AI 标记，人工最终判断 |
| 审批路由 | ✅ 自动 | 规则清晰，可配置 |
| 工单填写 / 摘要生成 | ✅ 自动（人工确认后提交） | 高效，但需人工确认内容 |
| 链上状态 / 余额查询 | ✅ 自动 | 只读操作，零风险 |
| **发起链上支付** | ❌ 必须人工确认 | 不可逆资金操作，必须授权 |
| **钱包权限变更** | ❌ 必须人工 + 多签 | 安全边界，不允许自动执行 |
| **争议处理 / 退款** | ❌ 必须人工 + 业务流程 | 涉及合同和法律效力 |

### 验证方式

- 链上 tx 记录：每笔支付有可独立核查的 hash
- Agent 操作日志：可审计每个决策的输入和输出
- 业务关联映射：链上支付与业务系统记录的对应关系

### 主要风险

| 风险 | 描述 | 缓解方式 |
|------|------|---------|
| AI 理解偏差 | 上下文提取错误导致错误路由或错误金额 | 人工确认节点兜底，高金额双重确认 |
| 链上操作不可逆 | 支付发出后无法撤回 | 模拟执行先行，严格人工确认前置 |
| 权限蔓延 | Agent 积累越来越多的钱包授权 | 每次任务独立授权，任务完成后权限失效 |
| 钱包私钥风险 | Agent 接触私钥 / 签名能力 | MPC 钱包或 Smart Account，Agent 不直接持有私钥 |
| 合规不确定性 | 跨境加密支付监管差异 | 先在合规明确的场景和区域验证 |

## 四、项目初步 Proposal

**项目工作名**：`AgentPay` — 企业业务流程 AI Agent × Web3 支付结算

### 目标用户

需要处理**多步骤审批 + 链上支付**的企业财务/运营人员，尤其是：
- 跨境支付场景（传统银行慢、成本高）
- 需要可编程支付条件的场景（满足条件自动触发，但必须有授权边界）
- 需要多方可信审计记录的场景

### 真实场景

企业服务采购：供应商完成交付后，AI Agent 读取交付确认文件，自动生成支付工单，推送给财务审批，审批确认后在 policy 范围内触发稳定币支付，链上留存审计记录。

### 最小功能（MVP）

1. **业务上下文理解**：输入文档/表单，输出结构化支付请求（金额、收款方、条件、关联业务 ID）
2. **审批路由**：根据金额和类型自动路由，生成摘要推送给审批人
3. **钱包状态查询**：Agent 可实时查询企业钱包余额、历史支付、链上状态
4. **受控支付执行**：在预设 policy（金额上限、白名单地址、时间窗口）范围内，人工确认后发起链上支付

MVP 不包含：自动触发无人工确认的支付、跨链桥接

### 验证方式

- 用 3–5 个模拟业务场景跑通全流程
- 对比指标：处理时长（全人工 vs Agent 辅助）、工单准确率
- Testnet 上留存完整的支付记录和 Agent 操作日志

### 可能赛道

- Cobo Hackathon（Agent Wallet / Permission 方向）
- ETHGlobal AI × Web3 track
- Stripe / x402 相关 Hackathon（Machine Payment 方向）

### Week 3 下一步

- 接入真实 Web3 支付层（Testnet，稳定币）
- 设计 policy / Pact：Agent 可自动执行的边界
- 做一个 10 分钟可演示的完整 demo

## 六、主方向深挖包

### 典型场景

**企业服务采购付款（$50K USDC，跨境）**

1. 采购系统触发：供应商提交交付确认报告
2. Agent 读取报告：提取金额、收款地址、关联 PO 号、交付物清单
3. Agent 检测：金额与合同一致 ✅，收款地址在白名单 ✅，PO 状态已关闭 ✅
4. Agent 路由：金额 > $10K → 推送给 CFO 审批，附上 2 句摘要
5. CFO 在 Slack/系统确认 → 触发 policy 检查
6. Web3 层：在 Pact 范围内发起 USDC 转账（多签钱包，需 2/3 签名）
7. 链上结算完成，tx hash 写入业务系统，审计日志自动生成

### 反例：什么不是这个方向

> "用 AI 帮用户填写加密货币交易所的提款表单" — 没有业务上下文、审批流程、policy 控制、链上审计集成。这是普通 UI 自动化，不是企业级 AI × Web3。

> "给企业做一个 Web3 支付页面展示余额" — 没有 AI 处理流程，Web3 只是展示层。

### 关键风险（5 条）

1. **AI 上下文提取偏差** → 金额或收款方识别错误 → 缓解：严格的人工确认前置，金额展示清晰
2. **链上支付不可逆** → 发出后无法撤回 → 缓解：模拟执行先行，分级确认阈值
3. **私钥/签名权限暴露** → Agent 不应直接持有私钥 → 缓解：MPC 钱包 / Smart Account，Agent 只发起请求
4. **policy 设计不当** → 白名单漏洞或金额上限设置错误 → 缓解：policy 变更需独立审批，定期审计
5. **合规风险** → 跨境加密支付监管不统一 → 缓解：先在合规明确区域验证，法律意见先行

### 最小验证计划

**一周内可完成：**
- [ ] 3 个模拟业务场景，Agent 读取文档 → 生成支付工单 → 记录准确率
- [ ] Testnet 上跑通：Agent 查询钱包余额 → 人工确认 → 发起稳定币转账 → 链上记录
- [ ] 设计并实现一个最小 policy：收款白名单 + 单笔金额上限 + 时间窗口
- [ ] 对比时长：全人工流程 vs Agent 辅助流程

**验证指标：**
- 支付工单字段准确率 > 90%
- 处理时长缩短 > 50%
- 零未经人工确认的链上资金操作
